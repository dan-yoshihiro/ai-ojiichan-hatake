# SNS 運用で試してやめた施策・訂正した判断は何か — 失敗の全記録

*最終更新: 2026-05-13（2026-07-17 に運用の失敗／開発の失敗で再編・用語を平易化）*

> **TL;DR**: AI農業先生方式で採用しなかった施策・廃止した施策・過剰実装と判定したものを 7 件、包み隠さず開示。前半（A）は SNS 運用そのものの失敗——AB テスト中止（標本不足＋機会費用）/ ハッシュタグ数比較の廃止 / 鼓舞 20% ハード目標の廃止（未検証の通説を撤回）/ 投稿計画の衝突とカレンダー統合。後半（B）は AI と開発する中で訂正した判断——重い行動ログ機構の過剰実装回避 / 「歴史的瞬間」という誇張の訂正 / 評価枠組みの置き場所ミス。共通パターンは「AI が"もっともらしい"提案を出す → 人間が批判的に評価する → 過剰・誤判定・装飾を訂正する」。失敗の構造を残すことが学習データとしての価値を高めるという思想。

採用しなかった施策・廃止した施策・過剰実装と判定したものの記録。

成功事例だけ残すと「綺麗な軌跡」に見えるが、実態は判断ミスとリカバリの連続である。本記事はそれを包み隠さず開示する。前半は SNS を運用する人にそのまま役立つ失敗、後半は AI と一緒に開発する中で訂正した判断に分けた。

## A. SNS 運用で試してやめた施策

### 1. ABテスト「翌日への問いかけ」（中止）

**「効きそう」で始めた施策が、2週間で誤差に消えた。** 小さすぎる標本は、そもそも何も判定できない——それを学んだ。

#### 仮説

投稿末尾に翌日問いかけを加えると、読者の翌日再訪問・リプライ率が上がる。

#### 設計と結果

2週間・対照／処置 の偶奇日割り当てで実施。標本は各群 1ケタ件と少なく、エンゲージメント差は誤差範囲を出ず、むしろ対照群がやや高いケースもあった。具体数値は非公開。

#### 判定: 中止（追加テストもしない）

理由:
1. **標本不足**: 1ケタ件規模では craft 効果と偶発の判別が困難
2. **より強いシグナル（bookmark）の発見**: 機会費用の観点で弱い仮説に追加投資しない

#### 学び

- 「効きそう」な仮説でも標本不足で判定不能になる規模で実施すると判定の根拠が弱い
- より強いシグナルが見つかった時点で機会費用的に弱い仮説を切る判断が必要
- ただし「効果なし」を確定したわけではなく「**現スケールでは効果検出不能 + 機会費用優先で中止**」が正直な記述

### 2. ハッシュタグ数の効果比較（廃止）

**「予定」は「やった」ではない。** どのデータが集まれば判定できるかを決めずに始めた比較は、そのまま空中分解した。

#### 仮説

ハッシュタグ数を変えて比較すれば、デフォルト設定を変更すべきか判断できる。

#### 結果: 判定不能（同種別内比較に必要なデータが集まっていなかった）

#### 判定: 廃止（再着手しない）

craft 学習ループの方が本命と判断し、ハッシュタグ数の細かい比較に投じる投資対効果が低いと評価。

#### 学び

- 「やる予定」と「実施済」は別物。「予定」だけ TODO に並んでも判定対象にならない
- 施策設計時に「どのデータが集まれば判定可能か」を明確にしないと、施策自体が空転する

### 3. 「鼓舞コンテンツ 週2回・全体の 20%」のハード目標を廃止（2026-05-11）

**通説を5週間まるごと守らなかったのに、目標は20日前倒しで達成した。** 未検証のベストプラクティスは、自分の数字で疑うべきだった。

#### 仮説

外部ブログ記事の「40% entertainment / 30% education / **20% inspiration** / 10% promotion」というコンテンツ配分の推奨を取り入れ、「鼓舞コンテンツを週 2 回・全体の 20% 入れる」をハード目標として設定していた（運用初期）。

#### 設計と結果

- 出典: 個人ブログの推奨（信頼性: 中・本アカウントの規模・領域・期間での実測ではない）
- 検証期間: 約 5 週半・37 日間（2026-04-04 〜 2026-05-11）
- 全オリジナル投稿 57 本のうち「鼓舞 craft」明確含有: 0-2 本（~3.5% 以下）
- それでも 1 人 → 50 人達成（5 月末 50 人目標を 20 日前倒し）

#### 判定: ハード目標を廃止（2026-05-11）

5 週間「鼓舞 ~0%」運用にもかかわらず目標前倒し達成。**本アカウントでは鼓舞 20% は必須ではない**ことが実証された。

#### 学び

- **未検証の通説を盲信せず、自分のデータで判断する**
- リプ craft には自然に鼓舞要素（「楽しみじゃのう」「ええ歩みじゃ」「見守る」等）が混じる → 投稿型だけで測ると実態を過小評価する
- ハード目標を維持すると「無理に 2 本入れる」craft 質低下リスク（DifyPrompt の「おお！」偏重と同じ構造）
- 新方針: 「機会があれば取り入れる」水準に格下げ
  - 大きな失敗・課題を公開した時（読者と一緒に乗り越える craft）
  - 投稿テーマがネガティブに偏った時（バランス調整）
- 残課題: フォロー解除の追跡が未実装のため「鼓舞ゼロで定着 OK」は厳密には未検証。次段階（50→100 人）で定着の課題が顕在化したら鼓舞 craft を再検討する

### 4. カレンダー統合せず投稿計画が衝突しかけた件

**気づいたら、同じ10日間に投稿候補が渋滞していた。** 別々に積み上げた施策を束ねる「関所」が運用に無かった。

ある時点で、複数の文脈で並行して積み上げた投稿候補が同じ10日間に集中し、1日 2〜3 投稿想定でキャパ超えになりかけた。

#### 修正

統合カレンダーを作って 1日1投稿原則に揃え、craft 軸が重複するものは取り下げ・優先度の低いものは後ろ倒し・本数を縮小して整合させた。

具体的な施策名・スケジュールは内部資料として非公開。

#### 学び

- 各文脈で別々に積み上がった施策を統合する確認点が運用に組み込まれていなかった
- カレンダー統合は週次レビューで定期的にやるべき作業

## B. AI と開発する中で訂正した判断

### 5. 行動ログ完備の記録機構（Phase B）— 過剰実装と判定・延期

**フォロワー数十人に、1〜2人日の重厚なログ機構を作ろうとした。** 作る前の「今それ要る?」で止めた。

#### 提案内容

人間の行動を全て構造化ログとして残す厚い記録機構を Phase B として提案。実装には1〜2人日規模が必要と見積もり。

#### 判定: 将来検討・延期

判定根拠:
- フォロワー数十人規模に対して 1〜2人日の記録機構実装は **過剰（YAGNI: 今必要ないものは作らない）**
- 既存資産（対話履歴・コミットログ・TODO の `[x]` マーク等）で大半は代替可能
- 戦略マネージャー出力の履歴データもまだ蓄積前

#### 学び

- 「完全ループ」の理想に引っ張られて重厚な記録機構を組むのは過剰
- 小規模では既存資産で十分代替可能
- 「100% にする」より「13% でも回す」のほうが実験段階としては合理的
- 商業化段階で再検討（多テナント運用が現実化したタイミング）

### 6. 「歴史的な瞬間」と書いたコミットメッセージ — 訂正

**「歴史的成果」と書いた。冷静に見て、歴史的ではなかった。** AI の"盛り癖"を人間が削った記録。

5/5 の学習ループ検証成功時、コミットメッセージで「Phase A 完全閉鎖ループ実証」「歴史的成果」と書いた。

#### 訂正

これは誇張。客観的には：
- 世界初ではない（マーケティング自動化のループは何十年も前から存在）
- 業界・人類にとってのインパクトはない
- 個人プロジェクトとしての節目であって「歴史的」ではない

正確な記述: 「**1ヶ月積み上げた個人プロジェクトの自然な節目で、1日で全体が動いた**」程度。

#### 学び

- 「文学的衝動で結論を盛り上げる」のは Claude（AI）の癖
- ユーザーから「賢いAIがバカっぽくなる」と指摘された
- 装飾語（「歴史的」「完全」「実証」「決定的」「致命的」「核心」「圧倒的」など）を意図的に避ける必要がある
- 事実 + 数字 + 具体名で書く。比喩や修辞を最小限に

### 7. 評価フレームワークを memory に書いた件

**戦略級の知見を、git に残らない場所に書いていた。** 「どこに残すか」を間違えると、その知見は誰にも届かない。

5/5 の対話で、3層評価フレームワーク（実行コスト削減・ネタ枯れ防止・完全ループ閉鎖）を Claude の memory に書いた。

#### 訂正

これは置き場所の判断ミス。アーキテクチャ／戦略に関わる知見は **project docs（git管理）** に置くべきだった。memory に書くと:
- git で版管理されない
- ユーザー以外見えない
- project doc を読む人が気づかない

#### 修正

project docs に正本として移動。memory には「自戒」だけ残した。

#### 学び

- 「memory に残す」と発言しただけで実際には書いていなかったケースが複数あった
- 「残す」と言ったらその場で実行する必要がある
- 知見の置き場所は「git に残るべきか」で判断する原則を確立

## 共通する学び

これらの失敗・修正に共通するパターン：

1. **AI（Claude）が「もっともらしい」提案を出す**
2. **ユーザーがそれを批判的に評価する**
3. **過剰・誤判定・装飾を訂正する**
4. **修正後の判断が project doc に残る**

判断軸が機能していたから過剰実装や誤った評価軸を避けられた。AI 任せにせず、ユーザーが疑う習慣を維持することがプロジェクトの精度を保つ鍵となっている。
